## Context

---

The client utilizes a proprietary product which manages A&P execution within one of its three existing modules. Originally developed internally a decade ago, this system now requires modernization to align with the company's strategic objectives. The application operates on Node.js, Express, AngularJS, Oracle DB, and Linux. The application today is a monolithic code with approximately 230k lines of code (150k backend / 80k frontend).

## Recommendation

---

We recommend to adopt **NestJS + Drizzle/Prisma + Better Auth** for the A&P backend modernization.

It provides the optimal balance for breaking down the 150k-line monolith into manageable modules while leveraging existing Node.js expertise. NestJS's enterprise-grade architecture with built-in dependency injection, guards, and modules prevents recreating the same maintainability issues, while its comprehensive tooling and TypeScript-first approach significantly improve development velocity. The framework offers a structured migration path that balances innovation with stability, enabling future microservices evolution when needed.

## Appendices

---

- Existing auth system :

    The backend uses JWT (JSON Web Tokens) with a hybrid authentication approach:

    Authentication System Components:

    1. JWT-based Authentication (middlewares/auth.js:5, routes/users.js:4)
    - Uses jsonwebtoken library for token generation and verification
    - Tokens are signed with HMAC SHA256 algorithm
    - Tokens are base64 encoded for transport
    2. External LDAP Authentication (routes/users.js:256-287)
    - Primary authentication through the client's external API:
    `https://api.client.io/v2/authentication/authenticate`
    - Falls back to LDAP validation for user credentials
    3. Bypass Mechanisms (routes/users.js:250)
    4. Token Flow:
    - Client sends email/password to /login endpoint
    - System validates against external LDAP service
    - On success, generates JWT with user email and ID
    - JWT is base64 encoded and returned to client
    - Subsequent requests require Authorization: Bearer <token> header
    5. Authorization Middleware (middlewares/auth.js:34-121)
    - Validates JWT on all protected endpoints
    - Extracts user info from decoded token
    - Requires company context (x-current-company header)
    - Multiple ignored URLs for public endpoints

    Frontend Token Handling :

    1. Token Storage (auth.controller.js:108):
    $window.sessionStorage["imm-usr-tkn"] = result.token;
    2. Automatic Header Injection (restclient.service.js:23):
    'Authorization': 'Bearer ' + $window.sessionStorage.getItem('imm-usr-tkn')
    3. Additional Context Headers (restclient.service.js:24-28):
    - X-CURRENT-PROFILE: User profile ID
    - X-CURRENT-COMPANY: Company context
    - X-CURRENT-LANGUAGE: User language
    - X-USER-ID: User identifier
    - X-APP: Application context
    1. Token Persistence:
    - Stored in sessionStorage (cleared on tab close)
    - No automatic refresh mechanism
    - SSO support via URL token parameter (auth.controller.js:274)

    Key Characteristics:

    - Not standard JWT: The base64 encoding layer makes it non-standard
    - Session-based: Tokens stored in sessionStorage, not localStorage
    - Rich context: Multiple headers provide user/company/app context
    - Manual management: No automatic token refresh or expiration handling
    - SSO integration: Supports external token-based authentication
